# Дизайн: Вход по QR-коду (issue #1, пункт #1)

**Дата:** 2026-06-06
**Статус:** утверждён, готов к реализации
**Зависит от:** ничего (но переиспускает `App.pending_proxy` из фичи #3)

## Цель

Дать пользователю альтернативный способ входа — отсканировать QR-код в
официальном приложении Telegram вместо ввода номера телефона и кода из СМС.
Это удобство онбординга (особенно когда номер набирать неудобно).

**Важно (honest-ограничение):** QR-вход НЕ обходит блокировки — он идёт через
те же серверы Telegram, поэтому VPN/прокси при блокировке всё равно нужен. При
включённом 2FA пароль всё равно спросят. Это удобство, не «вход без VPN».

## Решения (зафиксированы с пользователем)

1. **Отрисовка QR:** самописный генератор + рендер на Tk Canvas. Ноль новых
   зависимостей (нет риска PyInstaller / GPL — согласуется с политикой проекта,
   ср. удаление pystray/tkcalendar).
2. **Размещение:** переключатель «По номеру / QR» на экране входа (LoginView).
   Первичный логин. НЕ дублируем в «Добавить аккаунт» (scope-контроль).
3. **Истечение QR:** кнопка «Обновить» (не авто-recreate) — проще цикл поллинга.

## Архитектура

Четыре единицы с чёткими границами:

| Компонент | Файл | Ответственность | Зависит от |
| --- | --- | --- | --- |
| QR-генератор | `tg_exporter/core/qrcode_gen.py` (новый) | `qr_matrix(data) -> list[list[bool]]` — кодирует строку в QR-матрицу (byte-mode, уровень L, версии 1–10, авто-выбор маски). Чистая функция, ноль зависимостей. | — |
| QR-виджет | `tg_exporter/ui/components/qr_widget.py` (новый) | `QRCodeWidget(master)` на `CTkCanvas`: `set_data(url)` рисует матрицу чёрными квадратами на белом фоне с quiet-zone; `clear()`. | qrcode_gen, theme |
| QR auth-методы | `tg_exporter/core/auth.py` (правка) | `start_qr() -> QRLogin`; `poll_qr(qr, timeout) -> AuthResult` со step ∈ {SUCCESS, WAITING, PASSWORD_REQUIRED, EXPIRED}. Знание Telethon остаётся в core. | Telethon |
| LoginView QR-режим | `tg_exporter/ui/views/login_view.py` + `app.py` (правки) | Переключатель режима, показ QR, фоновый поллинг через worker, кнопка «Обновить», переход к 2FA, события. | всё выше |

### Поток данных

```
LoginView [клик «Войти по QR»]
  → App.start_qr_login() → worker.submit(_bg_qr_login)
[worker]
  client.connect()              # через pending_proxy, как обычный вход
  qr = auth.start_qr()          → событие qr_ready(url)
  loop (с проверкой флага отмены):
    res = auth.poll_qr(qr, timeout=5)   # короткий wait, не блокируем навсегда
    SUCCESS            → login_success  (далее как обычный вход)
    PASSWORD_REQUIRED  → qr_2fa         (показать поле пароля 2FA)
    EXPIRED            → qr_expired     (кнопка «Обновить»)
    WAITING            → continue
[UI каждые 80мс]
  qr_ready   → QRCodeWidget.set_data(url)
  qr_2fa     → показать _pwd_frame, кнопка «Войти» → auth.verify_password
  qr_expired → текст «Код устарел» + кнопка «Обновить» (recreate)
```

### Новые UIEvent

| event_type | payload | Источник |
| --- | --- | --- |
| `qr_ready` | `url: str` | `App._bg_qr_login` |
| `qr_2fa` | `None` | `App._bg_qr_login` |
| `qr_expired` | `None` | `App._bg_qr_login` |
| `qr_error` | `str` | `App._bg_qr_login` |

(`login_success` переиспользуется — далее как обычный вход.)

## QR-генератор — детали

- **Только byte-mode, уровень коррекции L** (максимум данных). YAGNI: не делаем
  numeric/alphanumeric/kanji и уровни M/Q/H — для QR-логина не нужны.
- **Версии 1–10.** URL Telegram = `tg://login?token=<base64url>` ≈ 45–65 символов
  → влезает в версию 4–5. Запас до 10. Таблицы (capacity, alignment positions)
  на 10 версий, не на 40.
- **Детекция истечения в `poll_qr`:** сравниваем текущее время с `qr.expires`
  (datetime, который отдаёт Telethon). Если `wait(timeout)` вернул без логина И
  `now >= qr.expires` → step=EXPIRED. Иначе (просто ещё не отсканировали) →
  WAITING. Это однозначное правило, без догадок по типу исключения.

- **Этапы:** выбор версии по длине → данные+EC (Reed-Solomon в GF(256)) →
  размещение по зигзагу + finder/timing/alignment паттерны → 8 масок, выбор по
  penalty → format-info.
- **Риск:** RS-арифметика и маски легко содержат тонкие баги, невидимые глазом.
  **Митигация — TDD против эталона:** разово запускаем проверенную реализацию
  (`qrcode` в одноразовом окружении, НЕ в зависимостях проекта) → генерим
  фикстуры-матрицы для нескольких строк → закоммитим фикстуры → наш генератор
  обязан воспроизвести их бит-в-бит. Плюс структурные инварианты (размер,
  finder-паттерны).

## Обработка ошибок

| Случай | Поведение |
| --- | --- |
| 2FA включён | `poll_qr` ловит `SessionPasswordNeededError` → `PASSWORD_REQUIRED` → поле пароля (переиспускаем `_pwd_frame`) → `auth.verify_password()`. |
| QR истёк | `qr_expired` → «Код устарел» + кнопка «Обновить» (`recreate` → новый QR). |
| Нет API ключей | Кнопка «Войти по QR» disabled (как phone-вход). |
| Уход с экрана / отмена | Флаг отмены в worker-цикле; переключение на phone-режим останавливает поллинг. |
| Сетевая ошибка / прокси не работает | `connect()` падает → `qr_error` → `_friendly()`. |

После успеха — то же, что обычный вход: `save_active_profile_session()`
(создаёт профиль, переносит `pending_proxy`).

## Тестирование

- `core/qrcode_gen.py` — TDD против эталонных фикстур (бит-в-бит) + структурные
  инварианты (размер по версии, finder-паттерны 7×7 в углах).
- `core/auth.py` QR-методы — юнит-тесты с замоканным `QRLogin` (все ветки
  `poll_qr`: SUCCESS / WAITING / PASSWORD_REQUIRED / EXPIRED).
- QR-виджет + LoginView — Tk нельзя запускать в фоне (SIGSEGV на teardown);
  проверяем компиляцией + ручной проверкой чистой логики (переключение режимов),
  как делалось для ProxyModal.

## Вне scope (YAGNI)

- QR-вход в «Добавить аккаунт» (только первичный логин).
- Авто-recreate по таймеру (выбрана ручная кнопка «Обновить»).
- Режимы numeric/alphanumeric, уровни коррекции M/Q/H.
- `ignored_ids` (защита от двойного входа тем же аккаунтом).
